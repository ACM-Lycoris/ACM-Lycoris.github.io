## 题目描述

<div class="qx-desc">

**天才 ACM**（洛谷 P10455 / AcWing 109）

给定一个整数 $M$，对于一个含有 $n$ 个整数的数列 $A$，我们定义数列 $A$ 的**校验值**如下：

对数列 $A$ 进行排序，然后从中取出**最多 $M$ 对数**（即最多 $2M$ 个数），使得每一对数**差的平方之和最大**，这个最大值就称为数列 $A$ 的校验值。

现在给定一个长度为 $n$ 的数列 $A$，以及一个整数 $T$。我们要把 $A$ **从左到右划分成若干段**，使得**每一段的校验值都不超过 $T$**。求**最少**需要划分成几段。

**输入格式：**
第一行输入一个整数 $K$，代表有 $K$ 组测试数据。对于每组数据，第一行三个整数 $n$、$M$、$T$；第二行 $n$ 个整数，描述数列 $A$。

**输出格式：**
对每组数据输出一个整数，即最少划分的段数。

| 范围 | 取值 |
|:---:|:---:|
| $n$ | $\le 5 \times 10^5$ |
| $M$ | $\le n$ |
| 数列元素、$T$ | 在 long long 范围内 |

</div>

## 先把题目读懂：什么是「校验值」？

题目里最绕的就是「校验值」这个定义。我们先把它单独拎出来弄明白，否则后面寸步难行。

给你一段数，校验值的算法是：

1. 把这段数**从小到大排序**；
2. 从中挑出最多 $M$ 对（也就是最多 $2M$ 个数）；
3. 每一对算它俩**差的平方**；
4. 让这些平方和**尽可能大**，这个最大值就是校验值。

关键问题来了：**怎么挑这 $M$ 对，才能让平方和最大？**

### 贪心结论：最大配最小

答案非常直觉：**排序后，用最大的配最小的，第二大的配第二小的，依此类推。**

举个例子。设排序后的数列是 $[1, 2, 5, 8, 8]$，$M = 2$（最多取 2 对）。按"最大配最小"：

- 第 1 对：最小 $1$ 配最大 $8$ → $(8-1)^2 = 49$
- 第 2 对：次小 $2$ 配次大 $8$ → $(8-2)^2 = 36$

校验值 $= 49 + 36 = 85$。

为什么这样配最优？直觉上，**平方差对"距离"特别敏感**——距离拉得越开，平方涨得越凶（$7^2=49$ 远大于两个 $3.5^2$ 之和）。所以我们要制造尽可能大的"贫富差距"：让队伍里最穷的去配最富的，把每一对都拉到最极端。如果你把两个相近的数配一起，差很小，平方更小，纯属浪费一个名额。

> 严格证明可以用**邻项交换法**：假设存在一种更优的配法没有"最大配最小"，那么总能通过交换其中两对，使平方和不减——反复交换最终必然收敛到"最大配最小"。这里不展开，记住结论即可。

### 对应到代码里的 `check`

来看代码里判断"一段数校验值是否 $\le k$"的函数：

```cpp
bool check(const vector<ll> &a, ll m, ll k)
{
    ll sum = 0;
    ll len = a.size();
    for (ll i = 0; i < min((len / 2), m); i++)
    {
        sum += (a[len - 1 - i] - a[i]) * (a[len - 1 - i] - a[i]);
        if (sum > k)
        {
            return false;     // 边累加边判断，超了立刻返回
        }
    }
    return true;
}
```

注意**传进来的 `a` 已经是排好序的**。循环里 `a[i]` 是第 $i$ 小，`a[len-1-i]` 是第 $i$ 大——这正是"最大配最小、次大配次小"。

- 上界为什么是 `min(len/2, m)`？因为最多取 $M$ 对，但如果这段数本身不够 $2M$ 个，最多也只能凑出 $\lfloor len/2 \rfloor$ 对，取两者较小值。
- `if (sum > k) return false;`：一旦累加超过 $k$ 就立即停手，省去后续计算。

搞定校验值，我们进入正题：怎么分段。

## 思路分析

### 第一步：贪心——每一段都尽量长

我们要让段数最少。一个非常自然的贪心想法是：

> 从左端点 $l$ 出发，**让这一段尽可能往右延伸**，直到再加一个数校验值就会超过 $T$ 为止。然后从断点后面重新开一段，重复这个过程。

这个贪心为什么对？关键在于一个性质：

> **校验值随区间扩大单调不减。** 一段数越长，能挑的"极端对"只会更多更狠，校验值绝不会变小。

所以对固定的左端点 $l$，合法性是"前缀封闭"的：如果 $[l, r]$ 合法，那么 $[l, r-1]$ 一定也合法。也就是说，存在一个**最大的 $r$** 使得 $[l, r]$ 恰好合法，再往右一个就爆。

既然如此，第一段贪心取到最长，剩下的子问题规模最小，绝不会比"第一段取短一点"更差（少切一刀总不亏）。这就是经典的贪心交换论证。

于是问题归结为一个核心操作：**给定左端点 $l$，快速找出最大的合法右端点 $r$。**

### 第二步：怎么找这个最大的 $r$？倍增登场

最朴素的办法是从 $l$ 一个一个往右试，$r = l, l+1, l+2, \ldots$ 直到爆掉。但如果每一段都很长，这就退化成 $O(n)$ 一段，配上每次的校验计算会非常慢。

由于合法性是单调的（前缀封闭），我们完全可以用**二分**来找这个最大的 $r$。而本题用的是二分的近亲——**倍增（Doubling）**，它的好处是不需要预先知道右边界，能自适应地"先大步试探，再小步逼近"。

倍增找最大 $r$ 的套路是这样的（维护一个步长 `step`）：

1. 初始 `step = 1`，当前已确认的右端点 `r = l - 1`（空区间）。
2. 尝试把区间扩大 `step` 个：看 $[l, r + \text{step}]$ 是否合法。
   - **合法** → 这一步可以吃下，更新 `r += step`，并把 `step` **翻倍**（下次迈更大的步）；
   - **不合法** → 步子迈大了，把 `step` **减半**（缩小试探幅度）。
3. 直到 `step` 减为 $0$，此时的 `r` 就是最大的合法右端点。

直观理解：先用 $1, 2, 4, 8, \ldots$ 指数级地猛冲，快速逼近边界；一旦冲过头，就把步长砍半，小心翼翼地往前蹭。整个过程对每一段只需 $O(\log)$ 次试探，比一格一格挪快得多。

### 第三步：归并优化——别每次都重新排序

倍增解决了"试探次数"的问题，但每次 `check` 都要求传入的数组**有序**。如果每试探一次就把整个 $[l, r+\text{step}]$ 重新 `sort` 一遍，排序本身就成了新瓶颈。

代码用了一个漂亮的优化：**维护一个 `curSorted`，它始终是当前已确认区间 $[l, r]$ 的有序版本。** 每次往右扩展时：

1. 只把**新增的那一小段** `nums[r+1 .. nextR]` 单独排序（这段通常很短）；
2. 用 `merge` 把"已经有序的 `curSorted`"和"刚排好的新段"**归并**成一个新的有序数组（归并是 $O(\text{长度})$ 的线性操作，不带 log）；
3. 拿归并结果去 `check`。如果合法，就把它存为新的 `curSorted`；不合法就丢弃，`curSorted` 保持不变。

这样我们就**避免了对同一批元素反复排序**——每个元素只在它"第一次被加入"时参与一次排序，后续都靠廉价的归并维护有序性。

## 手动模拟：跑一遍倍增

光说不练假把式。我们拿一个具体例子，把倍增的"翻倍—减半"过程完整跑一遍。

设数列 $A = [5, 1, 4, 8, 6, 2, 3]$（下标 $0 \sim 6$），$M = 1$，$T = 25$。

$M = 1$ 时校验值很简单：就是这段数的 $(\max - \min)^2$。

### 找第一段（从 $l = 0$ 出发）

初始 `r = -1`（空区间），`step = 1`，`curSorted = []`。

| 试探 | `step` | `nextR = r+step` | 候选区间内容（有序） | 校验值 $(\max-\min)^2$ | $\le 25$? | 动作 |
|:---:|:---:|:---:|:---:|:---:|:---:|:---|
| 1 | 1 | 0 | `[5]` | 0（不足一对） | ✓ | `r=0`，`step→2` |
| 2 | 2 | 2 | `[1,4,5]` | $(5-1)^2=16$ | ✓ | `r=2`，`step→4` |
| 3 | 4 | 6 | `[1,2,3,4,5,6,8]` | $(8-1)^2=49$ | ✗ | `step→2` |
| 4 | 2 | 4 | `[1,4,5,6,8]` | $(8-1)^2=49$ | ✗ | `step→1` |
| 5 | 1 | 3 | `[1,4,5,8]` | $(8-1)^2=49$ | ✗ | `step→0` |

`step` 归零，循环结束。第一段确定为 $[0, 2]$，即 `[5, 1, 4]`。

**看出门道了吗？** 罪魁祸首是下标 3 的元素 $8$：任何想越过它的扩展，都会把 $8$ 和最小值 $1$ 凑成一对，$(8-1)^2 = 49 > 25$ 直接爆掉。所以步长从 $4$ 一路减半到 $1$ 都失败，最终停在 $8$ 前面。这正是倍增"冲过头再退回来"的典型画面。

注意第 2 步翻倍后步长本应到 $4$，代码里还有一句 `step = min(step * 2, n - 1 - r)` 把步长**限制在剩余长度内**，防止越界空跑。

### 剩下的段

第一段切完后 `l = 3`，重复同样的过程：

- 从 $l = 3$：`[8]` ✓ → `[8,6]`（$(8-6)^2=4$）✓ → 再加下标 5 的 $2$ 变成 `[2,6,8]`（$(8-2)^2=36$）✗。第二段 = $[3,4]$ = `[8, 6]`。
- 从 $l = 5$：`[2]` ✓ → `[2,3]`（$(3-2)^2=1$）✓ → 到数组末尾。第三段 = $[5,6]$ = `[2, 3]`。

**最终划分成 3 段**：`[5,1,4] | [8,6] | [2,3]`，答案为 $3$。

## 代码实现

```cpp
#include <bits/stdc++.h>
using namespace std;
using ll = long long;
using ull = unsigned long long;

inline void optimizeIO()
{
    ios::sync_with_stdio(false);
    cin.tie(0);
    cout.tie(0);
}

// 计算有序数组 a 的校验值是否 <= k（最大配最小，最多 m 对）
bool check(const vector<ll> &a, ll m, ll k)
{
    ll sum = 0;
    ll len = a.size();
    for (ll i = 0; i < min((len / 2), m); i++)
    {
        sum += (a[len - 1 - i] - a[i]) * (a[len - 1 - i] - a[i]);
        if (sum > k)
        {
            return false;
        }
    }
    return true;
}

int main()
{
    optimizeIO();

    ll T;
    cin >> T;

    while (T--)
    {
        ll n, m, k;
        cin >> n >> m >> k;
        vector<ll> nums(n);
        for (ll i = 0; i < n; i++)
        {
            cin >> nums[i];
        }

        ll res = 0;
        ll l = 0;

        while (l < n)
        {
            ll r = l - 1;          // 先开一个虚空右边界（空区间）
            ll step = 1;
            vector<ll> curSorted;  // 始终是 [l, r] 的有序版本，初始为空

            while (step)
            {
                ll nextR = r + step;
                if (nextR < n)
                {
                    // 只对新增的一小段排序
                    vector<ll> nextPart(nums.begin() + r + 1, nums.begin() + nextR + 1);
                    sort(nextPart.begin(), nextPart.end());

                    // 归并：已有序的 curSorted + 新排好的 nextPart
                    vector<ll> Merge;
                    Merge.reserve(curSorted.size() + nextPart.size());
                    merge(curSorted.begin(), curSorted.end(),
                          nextPart.begin(), nextPart.end(),
                          back_inserter(Merge));

                    if (check(Merge, m, k))
                    {
                        // 合法：吃下这一步，步长翻倍
                        step = min(step * 2, n - 1 - r);
                        r = nextR;
                        curSorted = move(Merge);
                        continue;
                    }
                    else
                    {
                        step >>= 1;   // 迈大了，步长减半
                    }
                }
                else
                {
                    step >>= 1;       // 越界了，也减半
                }
            }
            // step 归零：r 即为以 l 为起点的最大合法右端点
            res++;
            l = r + 1;
        }

        cout << res << endl;
    }

    return 0;
}
```

### 几个值得停留的细节

**Q1：为什么 `r` 初始化为 `l - 1`？**

因为一开始这一段还是空的。空区间没有任何数对，校验值为 $0$，天然合法。`r = l - 1` 正好表示"区间 $[l, l-1]$ 为空"。

**Q2：`check` 里为什么传进来的数组一定有序？**

因为我们传的是 `Merge`，而 `Merge` 是 `curSorted`（已有序）和 `nextPart`（刚排序）归并的结果，归并本身就保证有序。`check` 依赖这个有序性来做"最大配最小"。

**Q3：不合法时为什么只把 `step` 减半，而不重置回 1？**

这正是倍增省时间的精髓。失败只说明"这一步迈得太大"，但比它小的步长可能仍然能成功。直接减半，相当于在已确认的 `r` 基础上做更精细的逼近，而不是从头来过。整个逼近过程是 $O(\log)$ 的。

**Q4：`step = min(step * 2, n - 1 - r)` 里的 `min` 有什么用？**

翻倍后步长可能超出数组剩余长度，导致后面 `nextR >= n` 不断空跑减半。用 `min` 把步长夹在剩余可用长度内，是个小小的剪枝。注意这里 `n - 1 - r` 用的是**更新前的旧 `r`**，把步长上限设为"到数组末尾还剩多少个"。

## 复杂度分析

设某一段最终长度为 $L$。

- **倍增的试探次数**：步长按 $1, 2, 4, \ldots$ 翻倍再减半，对每一段只有 $O(\log L)$ 次试探。
- **每次试探的代价**：排序新段 + 归并。归并是 $O(L)$ 的线性操作；排序新段的总长度（所有"被吃下"的新段加起来）不超过 $O(L)$，因此一段内排序总代价 $O(L \log L)$。
- **单段总代价**：$O(L \log L)$。
- **所有段加起来**：各段长度之和为 $n$，所以总复杂度约为
$$
\sum_{\text{段}} L \log L \;\le\; \log n \sum_{\text{段}} L \;=\; O(n \log n)。
$$

**对比朴素做法**：如果每次 `check` 都把整段 $[l, r]$ **重新排序**，单段就要 $O(L \log^2 L)$，总复杂度退化到 $O(n \log^2 n)$。归并优化的价值，就在于把"每个元素反复排序"变成"每个元素只排一次"，省下了一个 $\log$。

空间复杂度 $O(n)$，主要是 `nums` 和归并时的临时数组。

## 总结

天才 ACM 是一道把多个经典技巧串起来的好题，整条解题链路是：

1. **看懂校验值** → 贪心"最大配最小"，平方差对极端值最敏感；
2. **贪心分段** → 利用"校验值随区间单调不减"，每段都尽量取长，段数最少；
3. **倍增找右界** → 合法性前缀封闭，用翻倍—减半在 $O(\log)$ 内定位每段的最大右端点；
4. **归并维护有序** → 只排新增段并归并，避免反复排序，把复杂度从 $O(n\log^2 n)$ 压到 $O(n\log n)$。

一句话记住核心：**贪心决定"切在哪"，倍增负责"切多长"，归并保证"算得快"。**

同类的"倍增/二分 + 贪心分段"题型还有很多，比如各种"把序列分成最少段使每段满足某单调条件"的问题，掌握了这套"单调性 + 倍增逼近"的组合拳，遇到它们都能举一反三。
